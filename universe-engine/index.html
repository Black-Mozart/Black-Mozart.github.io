§ 01

The N-Body Gravitational Problem

Foundation

The core challenge: simulating a universe means computing the gravitational force between every pair of bodies at every timestep. With $N$ bodies that's $\tfrac{N(N-1)}{2}$ force evaluations — an O(N²) problem. At N=5000, that's ~12.5 million calculations per frame.

Newton's Law of Universal Gravitation
$$F = G \frac{m_1 \, m_2}{r^2}$$
Full N-body gravitational acceleration on body i
$$\vec{a}_i = G \sum_{\substack{j=1 \\ j \neq i}}^{N} \frac{m_j \,(\vec{r}_j - \vec{r}_i)}{|\vec{r}_j - \vec{r}_i|^3}$$
⚡

At N=5000 bodies, brute-force N-body requires ~25 million force pair evaluations per second at 60 FPS — far too slow. The Barnes-Hut tree reduces this to ~O(N log N), roughly 300k operations for the same N.

The gravitational constant $G$ in this engine is 500 px³ · s⁻² · mass⁻¹ — tuned to produce visually interesting orbital timescales at screen-space distances.

§ 02

Barnes-Hut Quadtree Algorithm

O(N log N)

The key insight: a group of distant stars can be approximated as a single point mass at their centre of mass. Barnes-Hut recursively divides 2D space into a quadtree. When computing force on a body, if a node is sufficiently far away, use the aggregate mass — otherwise recurse into its children.

Opening angle criterion (θ — Barnes-Hut condition)
$$\frac{s}{d} < \theta$$
where $s$ = cell width, $d$ = distance to cell's centre of mass
Centre of mass (incremental update as bodies are inserted)
$$\vec{r}_{cm} = \frac{\displaystyle\sum_{i} m_i \, \vec{r}_i}{\displaystyle\sum_{i} m_i}$$

🌲

This engine uses θ = 0.85 — a good balance between accuracy and speed. Lower θ (e.g. 0.5) = more accurate but slower. Higher θ = faster but less accurate. At θ=1.5 orbits become noticeably wrong; at θ=0.3 it approaches O(N²) again.

The tree is rebuilt from scratch every physics substep because bodies move each step. Build time is O(N log N); each query walk is O(log N) average.

§ 03

Gravitational Softening

Stability

When two bodies come very close, $r \to 0$ makes the gravitational force blow up to infinity, causing wild velocity spikes and numerical explosion. We add a softening parameter ε to regularise the potential at short range.

Standard (singular at r=0)
$$\vec{a}_i = G \sum_{j \neq i} \frac{m_j \,(\vec{r}_j - \vec{r}_i)}{|\vec{r}_j - \vec{r}_i|^3}$$
Softened (Plummer softening — stable for all r)
$$\vec{a}_i = G \sum_{j \neq i} \frac{m_j \,(\vec{r}_j - \vec{r}_i)}{\left(|\vec{r}_j - \vec{r}_i|^2 + \varepsilon^2\right)^{3/2}}$$
📐

This engine uses ε = 6 px. Softening also enters the BH tree query as dist² = dx²+dy² + ε². Physically, it smooths out the potential as if bodies had a finite size rather than being point masses — which they effectively are anyway, given their rendered radii.

The Plummer softening kernel corresponds to the gravitational potential of a Plummer sphere: $\Phi(r) = -\frac{Gm}{\sqrt{r^2 + \varepsilon^2}}$, which approaches the Newtonian $-Gm/r$ for $r \gg \varepsilon$.

§ 04

Velocity Verlet Integration

2nd-Order Symplectic

How do we advance positions and velocities forward in time? Simple Euler integration ($\vec{r} \mathrel{+}= \vec{v} \cdot h$) is first-order and dissipates energy — orbits slowly spiral inward or outward. Velocity Verlet uses the average of current and next accelerations, making it time-reversible and energy-conserving (symplectic).

Step 1 — Update positions using current velocity + old acceleration
$$\vec{r}(t+h) = \vec{r}(t) + \vec{v}(t)\cdot h + \tfrac{1}{2}\,\vec{a}(t)\cdot h^2$$
Step 2 — Rebuild BH tree, compute new accelerations at updated positions
$$\vec{a}(t+h) = \sum_j \text{BH-Force}\!\left(\vec{r}(t+h)\right)$$
Step 3 — Update velocities using average of old + new acceleration
$$\vec{v}(t+h) = \vec{v}(t) + \tfrac{1}{2}\left[\vec{a}(t) + \vec{a}(t+h)\right] h$$

Velocity Verlet is time-reversible: if you negate all velocities, the simulation runs backward exactly. This means no net energy injection or dissipation. The simulation runs 2 substeps per frame (each substep = dt/2) for additional accuracy.

§ 05

Collision Resolution & Fragmentation

O(N²) detection

Each pair of non-particle bodies is checked for overlap every physics substep. On overlap, the outcome depends on the relative speed: slow collisions merge (momentum conserved), high-energy impacts shatter both bodies into fragments.

Overlap detection — collision when
$$|\vec{r}_i - \vec{r}_j| < r_i + r_j$$
Momentum-conserving merge velocity
$$\vec{v}_{cm} = \frac{m_1 \vec{v}_1 + m_2 \vec{v}_2}{m_1 + m_2}$$
Fragmentation condition (both bodies must be large enough)
$$|\vec{v}_1 - \vec{v}_2| \geq v_{\text{thresh}} \quad \text{and} \quad m_1, m_2 \geq m_{\min}$$
Engine values: $v_{\text{thresh}} = 420\text{ px/s}$, $m_{\min} = 18$
Fragment mass allocation (3–5 fragments)
$$m_f = \frac{m_1 + m_2}{N_{\text{frags}}}, \qquad N_{\text{frags}} \in \{3, 4, 5\}$$
💥

Fragment velocities use alternating kick signs to ensure momentum is approximately conserved across the fragment cloud: odd-indexed fragments get +kick, even get -kick, so the net delta-momentum sums near zero around the centre-of-mass velocity.

§ 06

Shockwave Propagation

Radial Impulse

Supernovae emit an expanding shockwave ring that transfers impulse to any body caught in its wavefront. The wave expands radially at constant speed, decays in strength, and vanishes when it reaches its maximum radius.

Wave front radius over time
$$R(t) = R_0 + v_{\text{shock}} \cdot t, \qquad v_{\text{shock}} = 700 \text{ px/s}$$
Radial falloff function
$$f(R) = \text{clamp}\!\left(1 - \frac{R}{R_{\max}},\; 0,\; 1\right)$$
Impulse applied when body i is within wavefront thickness $w$
$$\Delta\vec{v}_i = \frac{S \cdot f(R)}{m_i + \delta}\;\hat{n}_i \qquad \text{when } |R - d_i| < w$$
where $S$ = wave strength, $\hat{n}_i$ = outward unit vector, $w$ = 14 px, $\delta$ = 5

The $m_i + \delta$ denominator ensures lighter bodies get a stronger kick (impulse = force × time, so lighter bodies accelerate more), while the $\delta$ prevents division by zero for near-zero mass particles.

§ 07

Black Hole Mechanics

Extreme Gravity

Black holes use supplemental accretion gravity on top of the BH tree force — a stronger, direct-sum force applied to every nearby body. Bodies that cross the event horizon are consumed.

Accretion pull (supplemental, applied every substep)
$$\Delta\vec{v}_i = \frac{1200 \cdot M_{BH}}{d^2} \cdot \hat{d} \cdot \Delta t$$
Event horizon — body consumed when
$$d < r_{BH} + r_{\text{body}} + 3$$
Schwarzschild-analogous radius
$$r_{BH} = 12 + 0.003 \cdot M_{BH}$$
Supernova remnant rules
$$M_{\text{star}} > 3200 \Rightarrow \text{Black Hole},\quad M_{rem} = 0.7\,M_{\text{star}}$$ $$M_{\text{star}} \leq 3200 \Rightarrow \text{Neutron Star},\quad M_{rem} = 0.35\,M_{\text{star}}$$
🌀

The bloom shader applies a gravitational lensing distortion near black holes: each black hole shifts UV coordinates toward itself with strength proportional to $M_{BH}/r^2$. This is a screen-space approximation — not ray-tracing — but creates a convincing visual warp in the bloom pass.

§ 08

System Performance Overview

Complexity

Each frame runs 2 physics substeps at $\Delta t / 2$ each. The dominant cost is the Barnes-Hut gravity kernel. Collision detection remains O(N²) — the next optimisation would be a sweep-and-prune broadphase.

Barnes-Hut Tree Build + Query O(N log N)
// ── TREE BUILD ──────────────────────────────────────────────────────
function buildTree(bodies):
  find bounding box of all non-particle bodies
  allocate root node  // covers bounding box, half-width = max(w,h)/2

  for each body b:
    insert(root, b)

function insert(node, b):
  // Incrementally update centre-of-mass
  newM       = node.totalMass + b.mass
  node.comX  = (node.comX * node.totalMass + b.x * b.mass) / newM
  node.comY  = (node.comY * node.totalMass + b.y * b.mass) / newM
  node.totalMass = newM

  if node is empty leaf:
    node.body = b                     // become occupied leaf
    return

  if node is occupied leaf:
    push existing body into correct child quadrant
    node.body = null                  // become internal node

  // Insert b into correct quadrant
  q = (b.x >= node.cx ? 1:0) + (b.y >= node.cy ? 2:0)  // 0=NW 1=NE 2=SW 3=SE
  if node.children[q] == null:
    allocate child with half-width = node.half / 2
  insert(node.children[q], b)

// ── FORCE QUERY (iterative BH walk) ────────────────────────────────
function queryAccel(px, py, queryIdx) -> vec2:
  stack = [root]
  accel = (0, 0)

  while stack not empty:
    node = stack.pop()
    if node.totalMass == 0: continue
    if node.body == queryIdx: continue     // skip self

    d = distance(px,py, node.comX, node.comY)
    s = node.half * 2                       // cell width

    if node.isLeaf OR (s / d) < θ:          // Barnes-Hut criterion
      accel += G * node.totalMass / (d² + ε²) * unitVec
    else:
      stack.push(all non-null children)    // recurse

  return accel
Velocity Verlet Integration O(N)
// ── VELOCITY VERLET (one substep) ──────────────────────────────────
function verletStep(dt):

  // Step 1: update positions using current velocity + half-step acceleration
  for each body i:
    pos[i] += vel[i] * dt + 0.5 * acc[i] * dt²

  // Step 2: rebuild BH tree at new positions, compute new accelerations
  buildTree(bodies)
  for each body i:
    newAcc[i] = queryAccel(pos[i].x, pos[i].y, i)

  // Step 3: velocity update using AVERAGE of old + new acceleration
  for each body i:
    vel[i] += 0.5 * (acc[i] + newAcc[i]) * dt
    acc[i] = newAcc[i]           // store for next step

// Compare with simple Euler (1st order, energy-dissipative):
function eulerStep(dt):
  acc[i]  = queryAccel(pos[i], i)
  vel[i] += acc[i] * dt          // uses old positions — energy drifts
  pos[i] += vel[i] * dt
Collision Resolution O(N²) detection
// ── COLLISION RESOLUTION ────────────────────────────────────────────
function handleCollisions():
  for i = 0 to bodyCount:
    if bodies[i].isParticle or isBlackHole: continue

    for j = i+1 to bodyCount:
      if bodies[j].isParticle or isBlackHole: continue

      d = distance(pos[i], pos[j])
      if d >= radius[i] + radius[j]: continue    // no overlap

      // Centre-of-mass frame quantities
      M    = mass[i] + mass[j]
      vCM  = (mass[i]*vel[i] + mass[j]*vel[j]) / M
      pCM  = (mass[i]*pos[i] + mass[j]*pos[j]) / M
      vRel = |vel[i] - vel[j]|

      if vRel >= FRAG_THRESHOLD and mass[i],mass[j] >= FRAG_MIN:
        // ── FRAGMENTATION ──
        nFrags = random(3, 5)
        for f in 0..nFrags:
          kick  = random(80, vRel * 0.6)
          sign  = f % 2 == 0 ? +1 : -1      // momentum balance
          spawn fragment at pCM with vel = vCM + spread * kick * sign
        remove bodies[i], bodies[j]
      else:
        // ── MERGE ──
        bodies[i].mass   = M
        bodies[i].vel    = vCM
        bodies[i].pos    = pCM
        bodies[i].radius = 2.5 + M^0.35 * 0.6    // mass-radius relation
        remove bodies[j]
Supernova Event Triggered
// ── SUPERNOVA ───────────────────────────────────────────────────────
function supernova(bodyIdx):
  star = bodies[bodyIdx]
  remove star from simulation

  // Emit expanding shockwave
  addShockwave(star.pos, strength = 1500 + star.mass * 0.7)

  // Spawn debris cloud
  debrisCount = clamp(star.mass * 2.8, 300, 1200)
  for i in 0..debrisCount:
    spawnParticle(star.pos, speed=random(220,900), life=random(0.9,3))

  // Determine remnant
  if star.mass > 3200:
    spawnBlackHole(mass = star.mass * 0.7)   // massive → black hole
    spawnAccretionDisk(star.pos)
  else:
    spawnNeutronStar(mass = star.mass * 0.35)  // smaller → neutron star

⚙ Do We Need WebAssembly?

Short answer: not at this scale. JavaScript with typed arrays (Float32Array, Int32Array) and an iterative (non-recursive) BH query runs the full engine at 60 FPS for up to ~5000 bodies in modern browsers.

WebAssembly (compiled from C/Rust) would become worthwhile when:

  • N > 10,000 bodies — WASM BH tree traversal is ~5–10× faster than JS for tight numerical loops, primarily due to SIMD vectorisation and no GC pauses.
  • Collision broadphase — a sweep-and-prune or BVH tree for O(N log N) collision detection in WASM would handle dense simulations JS can't.
  • Multi-threading — WASM with SharedArrayBuffer and Web Workers can parallelise the BH force query across CPU cores (each body's query is independent).
  • Emscripten port — the original C/raylib code could be compiled directly to WASM via Emscripten with minimal modification, running the GPU bloom shader via WebGL.

The current JS implementation uses typed arrays stored in contiguous memory (SoA layout), the iterative BH walk avoids call-stack overflow, and the physics runs in a single tight loop — about as fast as pure JS can get without WASM.
